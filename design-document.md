## 一、项目选题与核心理念

### 1.1 选题动机

本课程设计要求完成一个综合性的 C 语言程序设计项目。选择制作游戏的原因：

1. **覆盖知识面广**：游戏涉及结构化数据（结构体）、内存管理、文件 I/O、时间控制、多模块协作等
2. **趣味性强**：游戏比传统管理系统更具交互性和可展示性
3. **挑战性适中**：命令行游戏无需图形引擎，聚焦于核心逻辑

### 1.2 核心设计理念

```
吸血鬼幸存者的"自动战斗+割草升级" × UNDERTALE的"怪物设计+决心主题" = 命令行终端呈现
```

| 来源 | 借鉴要素 |
|------|----------|
| **Vampire Survivors** | 自动攻击、波次生存、升级三选一、多武器系统、属性叠加 |
| **UNDERTALE** | 怪物角色（Froggit 等）、决心（Determination）复活机制、像素美学 |
| **Roguelike 传统** | 网格移动、永久死亡（无决心时）、随机升级选项 |

### 1.3 设计原则

- **即时反馈**：每一帧都有动画（劈砍旋转、弹幕飞行、受伤闪烁），让终端"活"起来
- **策略深度**：升级三选一 + 武器组合 + 走位策略，每次游玩体验不同
- **技术克制**：纯 C 语言 + 标准库，不依赖第三方引擎，充分锻炼底层能力
- **跨平台兼容**：通过条件编译同时支持 Windows 和 Unix 系统

设计要点：

- **状态机模式**：用 `GameState` 枚举 + `switch-case` 实现清晰的游戏流程控制

- **中断处理**：暂停状态可在游戏过程中随时切入/切出，保留完整游戏状态

### 2.3 主循环设计

```

            ⑩ 波次完成检查
            ⑪ 死亡检查（决心复活）
            ⑫ 渲染输出
            ⑬ 帧率控制（Sleep + 忙等）

```

## 三、核心系统设计

### 3.1 帧率无关的移动系统

**问题**：不同终端渲染性能差异巨大（15-240 FPS），若按"每帧移动1格"会导致速度极度不一致。

**解决方案**：基于时间的移动计时器

```c
// 移动间隔 = spd × (1000ms / 60)，如 spd=3 → 50ms
int move_interval = player.spd * 1000 / 60;

// 累积本帧实际耗时（ms）
player.move_timer += player.frame_ms;

// 时间累积够了才移动
while (player.move_timer >= move_interval) {
    player.move_timer -= move_interval;
    player_move(&player, dx, dy);
}
```

- `frame_ms` 由帧率控制循环精确测量，确保任何帧率下移动速度一致
- `spd` 属性越小越快（每N帧 → 基于时间的间隔ms），升级减 spd 加速

### 3.2 帧率自适应校准

**问题**：渲染帧耗时因终端大小、系统负载而异，固定 sleep 时间不可靠。

**解决方案**：运行时校准 + 动态调整

```c
// 校准阶段：运行40帧测量实际平均耗时
for (i = 0; i < 40; i++) {
    计时开始 → render_frame() → 计时结束 → 累加耗时
}
target_frame_s = 平均耗时 × 1.08;  // 留8%余量

// 运行时：EMA 平滑显示 FPS
display_ema = display_ema × 0.9 + 实际耗时 × 0.1;
current_fps = 1.0 / display_ema;
```

- 校准上限 240FPS，下限 15FPS
- 忙等微调确保精确帧间隔
- Windows 使用 `timeBeginPeriod(1)` 将系统定时器精度提升至 1ms

### 3.3 渲染性能优化

**核心问题**：终端游戏逐字符 `printf` 造成大量系统调用，闪烁严重。

**解决方案**：三级缓冲 + 单次输出

```
帧缓冲(frame_buffer[17][60]) → 渲染缓冲(render_buf[24KB]) → fwrite(stdout)
   ↑ 格子级绘制                    ↑ 拼接ANSI转义序列           ↑ 唯一的系统调用
```

具体优化措施：

1. **帧缓冲** (`frame_buffer`)：二维 char 数组，先绘制所有格子，再统一拼接
2. **渲染缓冲** (`render_buf`)：24KB 栈缓冲区，拼接 ANSI 转义序列 + 字符 + 颜色
3. **单次 `fwrite`**：整个画面只调用一次 `fwrite(stdout)`，消除逐字符 I/O 开销
4. **ANSII 风格缓存** (`style_buf[256]`)：预计算每种字符的 ANSI 颜色包装，避免 `snprintf`
5. **全屏覆盖 (`\033[2J\033[H`)**：用 ANSI 清屏+光标复位，而非逐行打印空行
6. **敌人网格索引** (`enemy_grid[ARENA_H][ARENA_W]`)：O(1) 查找每格敌人，避免每格遍历 30 个敌人

### 3.4 武器系统设计

```
武器库 (weapon_library[5])
    │
    ├── WEAPON_STICK    [树枝]   近战  │
    ├── WEAPON_KNIFE    [小刀]   近战  ├── 瞬间伤害 + 劈砍动画
    ├── WEAPON_FIRE     [火焰]   近战  │   (effect_spawn_slash)
    │
    ├── WEAPON_BONE     [骨头]   远程  ├── 发射弹幕飞行命中
    └── WEAPON_LIGHTNING[闪电]   远程  │   (effect_spawn_projectile)
```

**近战 vs 远程的处理差异**：

| | 近战 | 远程 |
|------|------|------|
| 伤害时机 | 攻击瞬间 | 弹幕飞行到达后 |
| 视觉效果 | 劈砍符号旋转动画 (/ = \ !) | 飞行符号 (丨 ~) 追踪目标 |
| 命中方式 | 攻击时检查范围内最近N个敌人 | 弹幕每帧移动，到达目标位置命中 |
| 特殊机制 | AoE 溅射（火焰） | 穿透（骨头）、连锁多目标（闪电） |

### 3.5 升级选项生成算法

```c
// Fisher-Yates 洗牌 → 取前3个，保证不重复
1. 构建可用选项池（属性 + 未拥有武器 + 特殊）
2. 若池大小 ≤ 3，全部展示
3. 若池大小 > 3，Fisher-Yates 洗牌前3个
4. 返回3个随机不重复选项
```

- 治疗选项仅在 HP 不满时出现
- 已拥有武器不再出现
- 决心最多持有 2 层

### 3.6 敌人 AI 设计

```
每帧 enemy_update():
  for each active enemy:
    ① 受伤闪烁计时递减
    ② 移动计时器累加 → 达到 spd 阈值才移动
    ③ 方向计算：朝玩家方向 (dx, dy)
    ④ 随机偏移：20%概率只沿单轴移动（减少怪物聚集）
    ⑤ 碰撞检测：不与其他敌人重叠
    ⑥ 玩家碰撞：碰到玩家 → 造成伤害 + 弹开敌人
```

防御性设计：
- 敌人弹开后钳制在边界内
- 碰撞检测 O(n²)，但 n≤30 可接受

### 3.7 波次难度递进

```c
scale = 1.0 + (wave - 1) × 0.25  // 每波+25%
enemy.HP  *= scale
enemy.ATK *= scale
if (wave > 3) enemy.DEF += wave/3
if (wave > 5) enemy.SPD = max(8, SPD-4)
```

- Boss (Greater Dog) 每 5 波出现，额外 +wave×10 HP
- 生成间隔随波次缩短（188 帧 → 最短 60 帧）

## 四、数据结构设计

### 4.1 核心结构体

```c
Player {
    x, y           → 位置 (2D网格)
    hp, max_hp     → 生命值
    atk, def, spd  → 战斗属性
    level, exp     → 等级系统
    weapons[4]     → 最多4把武器ID
    weapon_cooldowns[4] → 各武器独立冷却
    determination  → 复活次数 (UNDERTALE核心机制)
    invincible     → 无敌帧计数器
    move_timer     → 时间累积移动
    frame_ms       → 上帧实际耗时
}

Enemy {
    x, y, hp, atk, def, spd → 基础属性
    type           → 7种UNDERTALE怪物
    symbol[8]      → UTF-8显示符号
    flash_timer    → 受伤闪烁
    active         → 对象池复用标志
}

VisualEffect {
    type           → PROJECTILE / SLASH
    x, y           → 当前位置
    damage, piercing, aoe_radius
    lifetime       → 剩余帧数（用于动画）
    target_idx     → 弹幕追踪目标
    symbol         → 显示字符（劈砍动画轮换）
}
```

### 4.2 对象池模式

敌人和效果系统都采用**对象池 (Object Pool)** 模式：

- `MAX_ENEMIES = 30`：预分配 30 个 Enemy 结构体
- `MAX_EFFECTS = 40`：预分配 40 个 VisualEffect 结构体
- `active` 字段标记使用状态
- 复用而非 malloc/free，避免内存碎片和分配延迟

## 五、跨平台设计

```c
#ifdef _WIN32
    #include <windows.h>
    #include <conio.h>
    // 使用 GetAsyncKeyState 实时按键检测（支持多键同时按下）
    // 使用 QueryPerformanceCounter 高精度计时
    // 使用 Sleep() 帧间隔
    // 使用 timeBeginPeriod(1) 提升定时器精度
    // UTF-8 控制台编码：SetConsoleOutputCP(65001)
#else
    #include <unistd.h>
    #include <termios.h>
    #include <fcntl.h>
    // 使用 termios 非阻塞字符读取
    // 使用 usleep() 帧间隔
#endif
```

| 平台 | 输入方式 | 计时方式 |
|------|----------|----------|
| **Windows** | `GetAsyncKeyState()` 实时轮询 + `_kbhit()` 清缓冲 | `QueryPerformanceCounter` 纳秒精度 |
| **Unix** | `termios` 非阻塞 + ANSI 转义序列解析 | `usleep()` 毫秒精度 |

## 七、技术亮点

1. **帧率无关移动**：基于 `frame_ms` 的时间累积，而非帧计数，保证不同终端下体验一致
2. **自适应帧率校准**：运行 40 帧测量实际渲染耗时，动态设定目标帧率
3. **三级缓冲渲染**：帧缓冲 → 渲染缓冲 → 单次 `fwrite`，消除闪烁和系统调用开销
4. **对象池复用**：敌人和特效预分配，`active` 标记管理，零运行时内存分配
5. **Fisher-Yates 洗牌**：升级选项随机不重复抽取
6. **切比雪夫距离**：8 方向网格移动使用 `max(|dx|, |dy|)`，符合直觉
7. **EMA 平滑显示**：FPS 显示使用指数移动平均，避免数值跳动
8. **条件编译**：同一套代码同时支持 Windows 和 Unix 终端

## 九、总结

Soul Survivor 是一个在 **命令行终端** 中实现的 **即时 Roguelike 生存游戏**。它融合了吸血鬼幸存者的自动战斗割草玩法和 UNDERTALE 的怪物/决心主题，在技术约束下实现了流畅的实时游戏体验。

项目在约 **2700 行 C 代码** 中实践了：

- **软件架构**：模块化设计、状态机模式、对象池管理
- **数据结构**：结构体嵌套、枚举类型、静态数组管理
- **算法**：Fisher-Yates 洗牌、冒泡排序、EMA 滤波
- **系统编程**：跨平台条件编译、高精度计时、内存缓冲优化
- **游戏设计**：数值平衡、难度递增、风险/回报决策

这是一个集趣味性与技术深度于一体的课程设计项目，体现了对 C 语言编程和游戏设计的综合理解。
